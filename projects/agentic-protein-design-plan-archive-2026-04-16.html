1. Objective Diagnosis

The earlier controller-centric plan was rigorous, auditable, and useful for replay analysis, but it was too narrow to capture the real opportunity opened by the empirical-software paper. The stronger object is not a one-step TTO router. It is a bounded search over executable inference software built around a frozen checkpoint.

Bottom line. The plan is now clear enough to support a stronger paper: freeze the generator checkpoint and evaluator, let the LLM mutate bounded local inference programs, inject ideas from literature and prior programs, execute each candidate under matched budget, and use tree search to improve the program archive over time.

What was strong
The earlier plan already had good discipline around budget matching, deterministic compilation, replay versus cold-start separation, and structured diagnostics.
What was too narrow
A planner that emits only the next operator leaves too much value on the table. It cannot express trajectory control, post-generation repair, cross-framework recombination, or target-specific orchestration logic.
Correct fixed boundary
Freeze the model checkpoint, budget, evaluation harness, and safety constraints. Do not freeze one specific codebase if that prevents meaningful inference-software search.
Correct mutable boundary
Mutate only bounded inference software around the checkpoint: scouting logic, trajectory branching, diagnostics, refinement calls, reranking, escalation rules, and stopping logic.
Main scientific opportunity
Recombination is real here. Complexa-native TTO, BindCraft-style post-generation repair, hallucination, and diffusion-trajectory control can be treated as reusable executable motifs rather than isolated baselines.
Main risk
If mutable code is unconstrained, compute will explode and attribution will collapse. The project therefore needs a bounded program API and a fixed matched-budget score before any serious search loop is run.
Strong

Local execution unit

The right node is still a local target-specific inference program. That keeps evaluation concrete and prevents the search object from becoming too vague.

Strong

Paper alignment

The plan now matches the empirical-software paper much more closely: idea injection, code mutation, sandbox execution, scoring, tree search, and backpropagation.

Needs discipline

Bounded mutable scope

The LLM should not edit arbitrary project code. It should write a constrained local program bundle that calls into a frozen model and an audited evaluator.

Needs discipline

Replay is no longer the main story

Replay remains useful, but it becomes a debugging benchmark. The main result should be matched-budget improvement from searched inference programs.

2. Revised Thesis and Boundary

The central thesis is now: protein binder design can be cast as a scorable search over local inference software built around a frozen protein-generation checkpoint. The LLM is not just a planner. It is a proposal, mutation, and recombination engine over bounded executable programs.

Core framing. The project should be described as fixed foundation model, mutable inference software. The search object is a local inference program. Cross-target transfer enters through reusable motifs and prior programs, not by replacing local execution with a single global policy.

Surface Fixed Mutable / searched
Foundation model Checkpoint weights, model I/O interface, reproducibility constraints None. The checkpoint is the experimental anchor.
Inference software Bounded API and allowed imports Scouting logic, trajectory branching, diagnostics, refinement calls, reranking, escalation, and stopping.
Auxiliary methods Allow-list of permitted external methods Whether and how to invoke BindCraft-style repair, hallucination-style rescue, physics reranking, or literature-derived trajectory ideas.
Evaluation Matched-budget evaluator, success thresholds, logging format None. Candidates compete under the same score definition.
Idea sources Literature retrieval stack, archive schema, expert-hint format Which ideas are injected into the next mutation prompt and how they are recombined.

Why this boundary is stronger

Freezing the checkpoint creates a fair scientific comparison while leaving enough freedom for the search to matter. If the codebase itself is treated as sacred, the search space becomes too narrow. If the foundation model is mutable, attribution collapses. Freezing the generator but searching over surrounding inference software is the right balance.

3. Intro and Contributions Draft

Draft Intro

Recent work has shown that large language models can search over program space by proposing, executing, and revising code under an automated score. We adapt this idea to protein binder design by treating inference-time control around a frozen protein-generation checkpoint as executable empirical software. Instead of asking an LLM only to choose the next operator, we ask it to write, mutate, and recombine local inference programs that allocate budget, control diffusion trajectories, invoke post-generation refinement, summarize diagnostics, and decide when to escalate or stop. Each candidate program is executed in a reproducible sandbox and scored by matched-budget biological outcomes. Tree search then uses these scores and structured failure metadata to decide which program to mutate next. This framing turns agentic protein design from a vague planning problem into a concrete, auditable search over inference software.

Draft Contributions
  • We formulate protein binder inference-time optimization as a scorable search over local inference programs built around a frozen checkpoint.
  • We introduce a bounded mutable program interface spanning scouting, trajectory control, diagnostics, refinement, reranking, escalation, and stopping.
  • We enable code-level mutation and recombination across reusable motifs drawn from Complexa-native TTO, BindCraft-style repair, hallucination, and literature-derived trajectory control.
  • We pair scalar program scores with a diagnostic sidecar that preserves geometry, sequence, physics, and redesign-limited failure modes during search.
  • We evaluate the system under replay, cold-start, and live matched-budget settings using unique successes, efficiency, regret, and ablations over recombination and retrieval.

Paper-level consequence. This is no longer just a routing paper. The main story becomes LLM-guided search over inference software, with replay accuracy treated as a secondary diagnostic rather than the main endpoint.

5. Search Object: Local Inference Program

The node in the search tree is an executable local inference program for one target. It is code, not prose. Transfer enters only through idea injection and motif reuse.

Bounded interface Role in the program Why it matters
prepare_target(...) Build target-conditioned context, preprocessing choices, and any initial target-specific priors. Keeps target handling explicit and reproducible.
generate_or_branch(...) Controls initial arms, branching schedule, restart logic, and trajectory exploration around the frozen generator. Captures the part of the search that config-only optimization cannot express well.
summarize_diagnostics(...) Compresses outputs into geometry, sequence, physics, redesign-gap, and runtime signals. Makes failure-mode-aware mutation possible.
refine_candidates(...) Optionally invokes BindCraft-style repair, hallucination-style rescue, or other allow-listed refinements. Where cross-framework recombination becomes concrete.
rerank_and_select(...) Filters and reranks candidates based on fixed score proxies, diversity, and diagnostic summaries. Lets the program express geometry-first or physics-aware selection logic.
decide_escalation(...) Chooses whether to widen search, run MCTS-like escalation, switch to refinement, or stop. Encodes orchestration as software rather than as a single prompt answer.
should_stop(...) Implements budget-aware stopping rules and expected-observable checks. Prevents uncontrolled budget burn.

What this keeps clean

The checkpoint remains frozen. The evaluator remains fixed. The program is local and executable. That means each node can still be scored fairly on one target, while mutation and recombination remain expressive enough to matter.

Expansion operators

Operator What it does Example
Mutation Rewrite a single current program to improve one stage of the local inference loop. Change trajectory branching or stopping logic.
Recombination Combine executable motifs from multiple parents or design traditions. Complexa scouting + BindCraft repair + hallucination rescue.
Retrieval-conditioned transfer Inject a motif that did well on a similar target or that appears in newly retrieved papers. Add a diffusion restart heuristic from literature.
Rescue patch Apply a tightly targeted code edit motivated by one dominant failure mode. Patch reranking after repeated geometry-open but selection-failed runs.

7. Human Design and Idea Injection

Human-designed

What experts still design

  • The scorable task, success thresholds, and matched-budget comparison protocol.
  • The bounded mutable API and import allow-list.
  • The frozen checkpoint and the fixed evaluator.
  • The initial seed program library and safety constraints.
  • The small set of expert hints that define the search space responsibly.
  • Manual inspection of top candidates to catch reward hacking or implausible shortcuts.
Injected ideas

What should enter the mutation prompt

  • Literature summaries about diffusion control, trajectory steering, interface repair, and binder refinement.
  • Prior successful programs and near-miss programs from related targets.
  • Complexa-native TTO motifs that already show value under this evaluator.
  • BindCraft-style post-generation repair or hallucination-style rescue motifs.
  • Short expert interventions of the form "if X is observed, try Y".
Idea source Representative motif How it helps the search
Complexa-native TTO Scout broadly, diagnose geometry, escalate only on evidence Provides high-quality native program structure and realistic operator combinations.
BindCraft-style methods Post-generation local repair or gradient-based interface cleanup Adds a strong refinement motif that is hard to recover from config-only search.
Hallucination / redesign Sequence-only rescue after geometry has opened Separates geometry acquisition from local sequence improvement.
Diffusion-trajectory literature Restart, branch, prune, or noise perturbation rules Introduces new in-trajectory control motifs that are naturally expressed as code.
Expert advice "Do not spend heavy refinement budget before a plausible interface family appears" Acts like the paper's problem-specific advice that can create score jumps when the search stalls.

8. Evaluation and Baselines

The main result should be matched-budget design improvement, not just replay agreement with a human decision trace. Replay remains useful, but it should be treated as a secondary debugging benchmark.

Objective assessment. The old plan over-weighted replay and cold-start routing quality because those were the easiest pieces to audit. The new framing is stronger only if the final paper is willing to center searched program quality under fixed compute as the main endpoint.

Benchmark mode Question answered Recommended use
Replay Can the system mutate or select a better next program from historical traces? Cheap debugging benchmark for search logic, not the main claim.
Cold-start Can the system propose a strong first local program from target context and literature alone? Important for generalization, but still secondary to live matched-budget outcomes.
Live search Does the searched inference program frontier beat strong static baselines under equal budget? Main scientific benchmark.

Primary metrics

  • Unique successes at matched budget: the main score.
  • Success per GPU-hour: efficiency view of the same search frontier.
  • Regret versus best known executable program: useful for replay and archive analysis.
  • Compiler validity and runtime robustness: necessary because the search object is code.
  • Diversity among successes: useful when many candidates collapse to one family.

Baselines and ablations

  • Static frozen-checkpoint default inference program.
  • Human-written local programs without LLM mutation.
  • Config-only search without code-level mutation.
  • No-recombination ablation.
  • No-literature / no-prior-program ablation.
  • Routing-only controller baseline.

9. Implementation Plan

Phase 0

Freeze the true boundary

Choose the checkpoint, fix the evaluator, define the budget unit, and finalize the allow-list of auxiliary methods.

Phase 1

Define the mutable program API

Lock the bounded local program interface before any mutation loop starts.

Phase 2

Seed the program library

Write a small hand-designed set of initial programs: conservative, geometry-opening, refinement-heavy, and trajectory-control variants.

Phase 3

Build the mutation harness

Prompt the LLM with code, diagnostics, and injected ideas; compile; execute; score; archive.

Phase 4

Add tree search

Track node visits, rank scores, diagnostic sidecars, and recombination parents; then run PUCT over the archive.

Phase 5

Run replay, then live search

Use replay for harness debugging and archive analysis, then move quickly to live matched-budget comparisons.

10. Current Status and Immediate Next Step

✓
The scientific boundary is now clear. Freeze the checkpoint and evaluator; search over bounded inference software.
✓
The replay-oriented controller work was not wasted. Its diagnostics, retrieval prototypes, and execution discipline become useful scaffolding inside the broader program-search system.
◐
The bounded program API still needs to be written down explicitly. That is the next implementation bottleneck.
◐
The current benchmark story is still too replay-heavy. The main result should shift to live matched-budget search over executable programs.
◐
The motif library is conceptually clear but not yet encoded. Complexa, BindCraft, hallucination, and literature-derived trajectory ideas should be captured as reusable promptable motifs.
!
Unconstrained code mutation should not be allowed. Without API and evaluator hardening, this project will look broad but not scientific.

Recommended immediate move. Do not spend the next week polishing replay prompts. Spend it freezing the mutable program boundary, writing the bounded local program API, and preparing the first seed program library for a real mutation-and-search harness.
