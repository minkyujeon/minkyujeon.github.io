ML-Intern + SCORE Plan for Autoresearch v2

Design memo for turning the current config-search version of Autoresearch v2 into a more complete test-time optimization pipeline for protein de novo design. The proposal keeps v2 as the SCORE-style optimizer kernel and uses ML-Intern as the research, orchestration, reporting, and tool layer.
Summary Trial Lessons ML-Intern SCORE JSON Examples Proposed Pipeline Trial and Error

Executive Summary

Core thesis
Do not replace autoresearch_v2 with ML-Intern. Keep autoresearch_v2 as the deterministic SCORE-style search kernel, and expose it to ML-Intern through a small protein TTO tool API.
ML-Intern role
High-level agent: reads papers, code, logs, run stores, and docs; proposes hypotheses; launches or resumes search; writes analysis reports.
SCORE role
Search method: PUCT over evaluated candidate artifacts, rank-score normalization, no rollouts, direct execute-and-score feedback.
Autoresearch v2 role
Domain optimizer: evolves ConfigOnlyInferenceProgram objects for Complexa inference and scores outputs with AF2-derived success criteria.
The trial-and-error page shows that the main blockers are integration and search-control problems: resource accounting for best-of-n/mcts, unbounded idea context, fragile tool payloads, and treating node-level ideas as a hard gate. The next design should address those before adding unrestricted code generation.

Trial-and-Error Carryover

The current trial-and-error page is the main source of engineering constraints for the next version. The key point is that the failures were not just "Qwen cannot produce ideas." They were mostly system-integration failures around serving, schema boundaries, resource accounting, and prompt-size control.

Observed root-cause familyWhat happened in v2 diagnosticsDesign requirement for ML-Intern + SCORE
Serving / environmentSome jobs failed before any scientific logic ran because the vLLM endpoint was not ready.ML-Intern should classify readiness failures separately from search failures and keep vLLM/Slurm logs attached to the run report.
LLM output parsingOpenAI-compatible local serving sometimes returned JSON arrays as strings or partially malformed arrays.Every LLM-generated JSON object needs strict schema validation, repair attempts, and failed-payload preservation.
Prompt/schema mismatchThe LLM could propose multi-stage fields while the executor only ran one stage.The tool surface must only expose executable actions. Unsupported fields should be rejected before GPU execution.
Node-level ideation gateEmpty node ideas stopped search, even though the config refiner could still use feedback and history.Ideas must be optional steering evidence, not a hard gate for child creation.
Logging schema driftA newly emitted fallback event crashed the loop because the event type had not been registered.Loop event schemas need tests, and unknown event labels should not kill an expensive run.
Resource accountingThe gate undercounted best-of-n and mcts cost, producing timeout-heavy BetV1 diagnostics.Use an active work multiplier before another full search: beam width, replicas, simulations, nsteps, batch size, and refinement cost.
Prompt-size controlThe refiner could see the full idea bank, and repeated evidence reads inflated Qwen context.Use bounded EvidencePack summaries and id-addressed artifacts instead of dumping the whole store into prompts.
Experiment designThere was an unresolved choice between generic deep seed profile and target-routed seeds.ML-Intern should record seed-profile policy as part of each task envelope and report when comparisons are not apples-to-apples.

Three Systems, Three Jobs

SystemGoalMain functionsWhat it should not own
ML-Intern Autonomous ML engineering assistant. Accepts user prompts, calls tools, reads files, searches docs/papers/GitHub, runs jobs, edits code, tracks sessions, asks for approvals, and reports results. The low-level PUCT state, fixed protein scoring contract, and deterministic experiment identity should remain outside the free-form chat context.
SCORE Inference-time scientific software search. Uses PUCT to pick candidates, asks an LLM to create variants, executes each candidate, scores it, backs up the score, and repeats under budget. Domain-specific biological validity and HPC job mechanics. SCORE is the search pattern, not the full protein-design system.
Autoresearch v2 Protein binder test-time optimization over a fixed Complexa backbone. Defines the evolvable JSON artifact, compiles it into Hydra overrides, executes Complexa, parses rewards CSVs, and stores scored tree nodes. Broad literature crawling, multi-repo implementation tasks, user-facing chat, and cross-project reporting are better handled by ML-Intern.

ML-Intern: How It Works

ML-Intern is a tool-calling agent. The CLI adds the user's prompt to a message history, builds OpenAI-format tool specifications, calls an LLM through LiteLLM, executes any returned tool calls, appends tool outputs to the context, and repeats until the LLM stops calling tools.

For a prompt like ml-intern --model openai/gpt-5.5 "your prompt", the quoted string is the task. The model decides whether to answer directly or call tools such as research, bash, read, hf_papers, hf_jobs, and GitHub/HF repo tools.

ML-Intern control loop

user prompt
  -> ContextManager.add_message()
  -> LiteLLM acompletion(
       messages=...,
       tools=...,
       tool_choice="auto")
  -> parse tool_calls[]
  -> ToolRouter.call_tool()
  -> append tool result
  -> repeat until no tool_calls

SCORE: Search Pattern

SCORE treats the scientific artifact as something an LLM can improve, then uses tree search to spend test-time compute on the most promising branches. In the original paper, the artifact is often code. In Autoresearch v2, the artifact is mostly a typed JSON inference program.

Candidate artifactCode in SCORE. ConfigOnlyInferenceProgram in v2.
LLM proposalGenerate mutation or recombination from parent, task description, and feedback.
ExecuteRun the candidate under a fixed harness. No rollout simulation.
Score and backupConvert raw metrics to a sortable rank key, store it on the node, increment visits, and repeat under budget.
SCORE ingredientAutoresearch v2 equivalentDesign note
PUCT tree searchPuctTree over program idsGood fit. The search state should remain durable and replayable.
Rank-score normalizationrank_key stores the sortable protein objective; PuctTree then converts scored nodes to normalized RankScore_T(u) for PUCT selection.The soft tie-breaker makes zero-success protein runs sortable before the SCORE-style rank normalization is applied.
Every node expandableAny saved program can receive another childFits an open-ended config/refinement action space.
No rolloutsEach child launches Complexa and gets scored from CSVsExpensive but scientifically grounded.
LLM code mutationLLM JSON config mutation, with optional gated Tier C patch laterSafer for protein TTO than unrestricted code generation.
RecombinationMerge two high-performing configs or idea bundlesShould be added explicitly as an operator with budget control.

Autoresearch v2: Current Artifact and Score

The current v2 artifact is ConfigOnlyInferenceProgram. It contains a target, one or more stages, search parameters, sampling parameters, reward weights, refinement settings, filters, warm-start references, lineage metadata, selected idea ids, and optionally a gated code patch.

The evaluator reads rewards_*.csv files and computes a primary success count. If post-hoc audited unique successes exist, ranking can use unique_successes; otherwise it falls back to n_successful.

Success criteria, simplified

row.sample_type == "final"
row.af2folding_i_pae < I_PAE_MAX
row.af2folding_plddt_log > PLDDT_MIN
sequence passes dedup/filter audit

rank_key =
  primary_successes
  + soft_tiebreaker(best_i_pae, best_plddt, n_finals)
Current trial-and-error root causes point to two immediate fixes before larger architectural work: replace beam-only resource validation with an active work multiplier that counts beam_width, best_of_n.replicas, and mcts.n_simulations; cap idea-bank context before the next clean 120-eval run.

Proposed Protein TTO Pipeline

The new system should be split into a high-level agent layer and a deterministic search kernel. ML-Intern should know how to ask for research, start or resume runs, inspect the tree, and prepare reports. Autoresearch v2 should remain responsible for schema validation, execution, scoring, deduplication, and PUCT state.

Concrete New Ideas

IdeaWhat to buildWhy it mattersFirst implementation target
Protein TTO tool for ML-Intern Add a built-in or MCP tool named protein_tto_search with operations init_task, run_budget, inspect_tree, resume, promote_candidate, and audit_best. Separates free-form chat from deterministic search state. ML-Intern can orchestrate without directly modifying the run store. A wrapper that shells out to python -m autoresearch_v2 and returns compact JSON summaries.
EvidencePack Define a compact JSON bundle with top programs, recent failures, timeout rates, idea performance, stderr labels, and target metadata. Prevents context overflow and avoids listing every stored idea in prompts. Extend idea_evidence.py to return capped, scored evidence packs.
Failure taxonomy classifier Classify each failed child into serving_error, schema_reject, timeout, oom, zero_success_near_miss, zero_success_bad_quality, or parser_drift. PUCT score alone does not tell the refiner whether to reduce compute, pivot search family, or change reward objective. Store failure_family in evaluation records and expose counts to the refiner prompt.
Active work multiplier Replace beam-only resource validation with a multiplier covering beam width, best-of-n replicas, MCTS simulations, branch count, nsteps, batch size, and refinement iterations. The trial-and-error page shows timeout-heavy BetV1 runs because non-beam cost was undercounted. Patch refiner.py validation before another expensive 120-eval run.
Idea-conditioned operators Treat ResearchIdea not as a hard gate, but as an optional operator that biases config mutation and recombination. Empty node ideas should never stop search; useful ideas should still steer search. Make selected_idea_ids a ranked context field and allow no_idea fallback.
Operator bandit Track success by operator family: mutate_config, recombine_config, inject_research_idea, warm_start, bounded_patch. A fixed operator ratio wastes compute when one family repeatedly times out or produces zero-success children. Add per-family counts and recent best delta to the tree state or run store.
Recombination for configs and ideas Pick parent A from PUCT and parent B from top-K or complementary idea clusters. Ask Qwen to compare, summarize, then produce a hybrid config. This recovers a central SCORE mechanism while staying within typed JSON. Add a recombine_programs(parent_a, parent_b) path in the refiner.
Multi-target promotion Search on a small dev target set, then promote candidates only after heldout target and unique-cluster audits. Protects against overfitting one target or gaming the AF2 reward threshold. Add dev_targets, holdout_targets, and promotion_policy to run config.
Bounded Tier C patch gate Allow source patches only after config/idea search plateaus, only in whitelisted search files, and only with A/B comparison against the same config. Preserves the SCORE code-improvement idea without turning protein TTO into uncontrolled codegen. Start with patch proposals disabled, then add dry-run/import/smoke checks.
Auto-updated public analysis pages Generate pages like trial-and-error.html, tree views, run summaries, and root-cause ledgers directly from run records. The current manual website is valuable but can lag behind run-store truth. Add a report command that emits static HTML from RunStoreV2.

Proposed Bridge API

The bridge should be intentionally small. ML-Intern gets high-level verbs; Autoresearch v2 keeps the detailed schema and state transitions.

{
  "tool_name": "protein_tto_search",
  "operations": {
    "init_task": {
      "inputs": ["targets", "budget_evals", "seed_profile", "llm_model", "constraints"],
      "output": "run_id, seed_program_ids, initial_evidence_pack"
    },
    "run_budget": {
      "inputs": ["run_id", "additional_evals", "max_parallel_workers"],
      "output": "run_summary, top_programs, failure_summary"
    },
    "inspect_tree": {
      "inputs": ["run_id", "top_k", "include_recent_failures"],
      "output": "compact tree snapshot"
    },
    "propose_ideas": {
      "inputs": ["run_id", "target", "evidence_pack", "max_ideas"],
      "output": "ResearchIdea[]"
    },
    "promote_candidate": {
      "inputs": ["run_id", "program_id", "holdout_targets", "replicates"],
      "output": "promotion audit report"
    },
    "export_report": {
      "inputs": ["run_id", "format"],
      "output": "html_path, json_path"
    }
  }
}
The bridge should return compact JSON or short text, never full raw trajectories. The full artifacts stay in the run store and are addressed by ids.

Scoring and Promotion Policy

Online tree score

{
  "primary_metric": "unique_successes_if_available_else_n_successful",
  "rank_key": "primary_successes + soft_tiebreaker",
  "tie_breakers": ["best_i_pae", "best_plddt", "n_finals"],
  "cost_fields": ["wall_time_minutes", "gpu_minutes", "timeout_count"]
}

Promotion score

{
  "required": [
    "heldout_targets_passed",
    "unique_successes_audited",
    "no_duplicate_sequence_mode",
    "resource_cost_within_budget"
  ],
  "reported": [
    "best_i_pae",
    "best_plddt",
    "pass_rate",
    "sequence_diversity",
    "failure_family_counts"
  ]
}

The online PUCT score should stay simple and stable. Cost should usually guide scheduling and promotion rather than replace the scientific objective. Changing the primary score mid-run makes historical nodes hard to compare.

Implementation Roadmap

PhaseWorkExit criteria
0. Stabilize current v2 Fix active work multiplier, cap idea context, add schema tests for new event types, preserve fallback-to-refiner behavior. A clean CD45/BetV1 diagnostic can reach its intended eval budget without being dominated by timeouts or prompt bloat.
1. Add bridge tool Implement protein_tto_search as a local ML-Intern tool or MCP server wrapping python -m autoresearch_v2. ML-Intern can initialize, run 2-5 dry-run or smoke evals, inspect the tree, and export a compact report.
2. EvidencePack and failure taxonomy Persist failure labels and return capped summaries for top, recent, zero-success, timeout, near-miss, and idea-performance groups. Refiner prompts no longer list the full idea bank or raw repeated tool outputs.
3. ResearchIdea integration Let ML-Intern research papers/logs and write candidate ResearchIdea records. The v2 refiner consumes only selected/capped ideas. Zero-success plateau causes an idea-conditioned pivot, not a compute-only retry loop.
4. SCORE recombination Add two-step compare-and-hybridize recombination for typed configs and idea bundles. Top-K branches can combine complementary settings without free-form code edits.
5. Parallel search and promotion Add virtual loss, job leases, GPU pool accounting, resume semantics, heldout promotion, and report generation. Multi-worker runs are reproducible, resumable, and auditable across dev and heldout targets.
6. Bounded Tier C Enable search-file patches only after plateau triggers and A/B gate checks. A patch can prove value under the same config baseline before becoming part of the search population.

Main Design Risks

RiskWhy it mattersMitigation
Sparse rewardsMany children score zero, so PUCT has weak signal.Use soft tie-breakers, near-miss labels, failure families, and idea-conditioned pivots.
Metric hackingA config can optimize AF2-facing metrics without producing robust binders.Use heldout targets, unique-cluster audits, sequence diversity, and fixed evaluator versioning.
Compute wasteEach child can cost real GPU time. Bad resource gates produce timeout-heavy trees.Active work multiplier, early budget caps, parallel leases, and timeout-aware refiner feedback.
Prompt bloatLong idea banks and tool outputs reduce Qwen reliability and latency.EvidencePack summaries, duplicate read suppression, top/recent caps, and ids for full artifacts.
Tool-call driftLocal OpenAI-compatible Qwen may return JSON strings or malformed arrays.Strict schema validation, retry with smaller JSON, defensive parser, and trajectory examples.
Uncontrolled codegenFree code edits can break reproducibility and biological assumptions.Keep Tier C disabled by default; whitelist files; require A/B smoke and rollback.
