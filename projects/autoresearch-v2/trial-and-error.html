Autoresearch v2 Trial and Error

Fact-focused record of issues found while turning the first v2 config-search version into a more robust Qwen-driven search loop. This page separates observed failures from design changes and avoids treating diagnostic partial runs as final experiments.

Current Conclusion

Primary lesson
Ideas should help the config refiner, but node-level ideation must not be a hard gate for running a child config.
Qwen context
The local checkpoint advertises max_position_embeddings=262144. Earlier full scripts limited vLLM to --max-model-len 65536; current full scripts default to 131072.
Observed vLLM capacity
A 64K server log reported about 446688 GPU KV-cache tokens, so longer single-request contexts should be possible, with lower concurrency and higher latency.
Design fix
Keep thinking enabled, raise context if needed, and also compact tool evidence so the model is not encouraged to read unbounded history.

Scope and Non-Claims

What this page is
A chronological engineering record of v2 debugging: observed failures, concrete patches, residual risks, and how CD45/BetV1 diagnostic jobs informed the next change.
What it is not
It is not a final benchmark table and does not claim that a partial diagnostic run is representative of v2 performance.
Main code path
IdeaResearcher proposes research ideas; ConfigRefiner turns those ideas plus feedback into config edits; ProgramExecutor runs one Complexa stage.
Current blocker
The clean rerun should wait for resource accounting for best-of-n/mcts and bounded idea context.
Numbers in this page are anchored to specific Slurm jobs or run-store artifacts at the stated verification time. Running jobs can move after the snapshot; that is why live BetV1 counts are labeled as partial artifacts.

Investigation Thread

Question during debuggingAnswer reachedEvidence used
Can v2 use Qwen for idea proposals, or is Qwen only doing tool calls? Qwen is used both to drive the IdeaResearcher tool loop and to produce final structured ideas. It is also used by the config refiner to propose edits. Successful ideation probe 7415465 generated four valid research ideas; later run stores accumulated many ResearchIdea objects.
Should Qwen thinking be disabled to save context? No. The decision was to keep thinking enabled because idea quality matters, then increase context and reduce repeated evidence instead. The observed overflow was near the 64K serving limit, not the model's advertised maximum context.
Is a larger context window alone enough? No. Larger context avoids hard limit errors, but it does not fix malformed tool payloads, hard-gating on empty ideas, unbounded idea-bank prompts, or expensive configs. After 131K full scripts, 7435022 failed on event schema and 7435023 showed timeout-heavy behavior.
Should empty node-level ideation cancel the search? No. Empty node ideas should fall back to the config refiner using parent feedback, history, and existing ideas. 7428512 and 7429369 stopped early despite successful Slurm completion because empty node ideas/proposal exhaustion halted useful expansion.
Is the current no-success pattern purely a target difficulty issue? No. CD45 had a near miss at i_pAE=0.18456006, but both targets were also affected by code/schema failures, timeout-heavy configs, and prompt/context design issues. CD45 7435022 failed from a Python schema exception. BetV1 7435023 had many 30-minute timeouts before reaching a clean 120-eval finish.
Is v2 ready for a full 120-eval result? Not yet. It can run pieces of the pipeline, but a clean full result should wait for resource accounting and bounded idea context. Latest CD45 failed at 15/120; latest BetV1 was still running at verification with 39 timeouts out of 63 evaluations.

Why Did IdeaResearcher Fail?

The failures were not best explained as "Qwen cannot generate ideas." Qwen did generate valid ideas in the 64K ideation probe, and it produced meaningful config-refinement actions, including resource-recovery proposals and a BetV1 child with one primary success in a partial diagnostic run.

The main failures were integration and prompting-design issues:

  • Context management: with thinking enabled, the OpenAI-compatible client reserves at least 8192 completion tokens. A server launched at --max-model-len 65536 therefore leaves roughly 57344 tokens for input. One trajectory reached that boundary because tool evidence was too verbose. The current full scripts use 131072.
  • Tool payload robustness: Qwen/vLLM sometimes returned a JSON list as a string, not as a native list. The prior parser treated that as zero ideas even when the content was mostly valid.
  • Search-loop design: node-level idea generation was treated as required. That made an empty idea list stop progress even though the refiner could still mutate from parent feedback and run history.
  • Schema mismatch: exposing multi-stage knobs before the executor supported them let the LLM propose configs that could not be faithfully executed.
  • Repeated evidence: exact duplicate tool calls are now skipped, but overlapping tools can still contain related information. Bounded summaries remain necessary.
Qwen still contributes operational sensitivity: long thinking traces, repeated evidence reads, and occasional schema-format drift are real behaviors. The current design should absorb those behaviors rather than assuming perfect tool calls.

Context-Length Policy

The local Qwen config supports up to 262144 positions. The 64K limit came from Slurm script launch arguments, not from the checkpoint config. Raising vLLM to 131072, 196608, or 262144 is technically plausible, but it changes runtime behavior:

  • Longer context lowers maximum concurrency because more KV cache is reserved per request.
  • Prefill latency grows with prompt length, so the LLM loop can become slower even if it does not OOM.
  • A larger window does not fix malformed tool payloads or hard-gating on empty ideas.
  • Unbounded evidence can hide the useful signal inside too much run history; compact tools are still needed.

Current Design Policy

  • Treat ideas as optional structured inspiration, not as a required gate for tree expansion.
  • Let ConfigRefiner propose from parent feedback, score history, stderr tails, and selected ideas when available.
  • Validate resource recovery before execution: batch, product, nsteps, checkpoints, refinement iterations, and MCTS simulations.
  • Parse tool outputs defensively because OpenAI-compatible local serving can vary the exact payload representation.
  • Use longer Qwen context when useful, but keep evidence summaries compact and bounded.
  • Cache exact repeated evidence reads inside one IdeaResearcher trajectory and clean that cache when the trajectory ends.
