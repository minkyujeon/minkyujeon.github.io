4. Supervisor: control decision and launch selection

The Supervisor receives decision-safe EvidencePack, parent portfolio, decision-safe analyst findings, and verifier results. It does not make new backend configs. It chooses one control action and, for launch, selects only verifier-launchable proposal ids.

Controller-level enforcement takes the intersection of Supervisor-selected ids and verifier_results[].launchable == true. A Supervisor-selected but verifier-rejected proposal is not launched.

Exact score definition and how the tree uses it

The code has two related but different quantities. Per-node score is a controller heuristic stored on each child node after that child finishes. The public run objective is cumulative strict global-unique successes across the whole run. They are intentionally not the same object.

quantitydefinitionwhere it is used
audited_unique_successNumber of unique binder sequences inside one child that pass the strict criterion: normalized i_pae <= 7/31, plddt >= 0.90, and binder_scrmsd_ca < 1.5.Primary per-child success count in EvalReport.metrics.
audited_unique_success_per_gpu_houraudited_unique_success / gpu_hours. This is a compute-normalized count, not a fraction within nsamples.Main primary term for node.score in default rate mode.
node.scorePer-child heuristic from score_eval_report(eval_report). It combines capped success/GPU-hour, soft structural signals, diversity bonus, and failure penalties.Stored on each ProgramNode; used directly by portfolio and puct, and kept for reporting/fallback even when puct_marginal is active.
strict_global_uniqueGlobally deduplicated strict-success binder sequences across all children in the run.Main y-axis on this page and Phase 6 objective. This is not simply the max child score.
puct_marginal QSubtree-level rate: strict successes from a node plus descendants, deduplicated by sequence when available, divided by subtree GPU-hours, plus recent marginal yield.Exploitation term for the puct_marginal parent picker.

2. Default score formula: AR22_SCORING_MODE=rate

primary_raw = audited_unique_success_per_gpu_hour
            # fallback: online_success_per_gpu_hour if audited value is missing

primary = min(primary_raw, AR22_PRIMARY_SUCCESS_RATE_CAP)
          # default cap = 2.0
          # set AR22_PRIMARY_SUCCESS_RATE_CAP=none/off/false to disable

interface_signal  = max(0, 0.60 - best_i_pae) * 0.30
confidence_signal = max(0, best_plddt - 0.90) * 0.10
rmsd_signal       = max(0, 1.5 - best_binder_scrmsd_ca) * 0.03
strict_bonus      = 0.10 if audited_unique_success > 0
                    or best_joint_candidate.strict_success is true
diversity_bonus   = max(0, 0.1 * (1 - duplicate_fraction))

penalty = -1.0 if failure_modes intersects {oom, timeout, backend_exception}
        + -0.3 if proxy_mismatch
        + -0.5 if status not in {completed, not_run}

node.score = round(primary + interface_signal + confidence_signal
                   + rmsd_signal + strict_bonus + diversity_bonus
                   + penalty, 6)

The strict bonus is intentionally tied to a paired strict candidate or audited success count. Axis-wise extrema from different rows do not by themselves count as a strict success.

3. Optional score formula: AR22_SCORING_MODE=smoothed_rate

count = audited_unique_success
        # fallback: online_success if audited count is missing

primary = count / (gpu_hours + AR22_SCORING_TAU)
          # default tau = 0.5 GPU-hours

node.score = round(primary + same soft bonuses and penalties, 6)

The running Phase 6 A/B jobs shown here are named by policy and calibration status; this page does not infer a scoring-mode environment override unless it is visible in artifacts. The code default is rate.

4. How each parent policy uses the score

policyexploitation termeffect
portfolioUses node.score directly for the exploit slot and backup ordering. Separate ranks choose iPAE-near-miss, pLDDT-near-miss, explore, and rescue slots.Structured categorical parent set: exploit + near-miss + explore + rescue.
puctQ(node) = node.scorePrioritizes high-scoring individual children while adding a UCB exploration bonus for less-expanded nodes.
puct_marginalQ(node) = subtree_audited / (subtree_gpu_hours + tau) + 0.5 * recent_marginal_yield_ratePrioritizes branches that keep adding globally new strict successes per Complexa GPU-hour; node.score is not the Q term here.

5. PUCT equations in tree-search terms

For puct:
    Q(node)   = node.score
    P         = 1 / N_completed              # uniform prior
    N_child   = len(node.children)           # how many direct expansions this node has
    N_total   = sum(len(n.children) for n in completed_nodes)
    UCB(node) = Q(node) + c_puct * P * sqrt(N_total) / (1 + N_child)

For puct_marginal:
    sub_aud(node) = |union audited_success_sequences over node + descendants|
    sub_gh(node)  = sum gpu_hours over node + descendants
    base          = sub_aud(node) / (sub_gh(node) + 0.5)

    recent_aud    = net-new strict sequences in the node's last 3 direct children
    recent_gh     = gpu_hours of those last 3 direct children
    recent_rate   = recent_aud / (recent_gh + 0.5)

    if node has >=3 children and recent_aud == 0:
        base = 0.5 * base

    Q(node)   = base + 0.5 * recent_rate
    UCB(node) = Q(node) + same exploration bonus

puct_marginal also has a diverse-family backup: if the last selected slot duplicates a family already represented in earlier slots, it may be replaced by the highest-UCB node from an unrepresented family.

6. TargetSpec in this implementation

There is no separate TargetSpec dataclass in current v2.2 code. In README/architecture language, TargetSpec means the RunConfig.target dictionary from the JSON config.

"target": {
  "id": "05_CD45",
  "difficulty_hint": "hard",
  "application_spec": {
    "modality": "binder",
    "epitope_policy": "backend_default",
    "mechanism": "unspecified"
  }
}

The controller places this target block into EvidencePack, uses target.id as the Complexa task name via ++generation.task_name=05_CD45, and treats target/threshold changes as immutable. Detailed target-chain, hotspot, and binder-length definitions come from the Proteina-Complexa target dictionary for that task.

PUCT vs puct_marginal: same tree input, different parent choice

Both policies receive the same completed SearchTree and use the same UCB exploration bonus. The factual difference is the exploitation term Q: puct trusts the score of one node, while puct_marginal asks whether that node's branch is still producing new global-unique successes.

puct

Q = node.score
UCB = Q + c_puct * P * sqrt(N_total) / (1 + N_child)

Interpretation:
pick high-scoring individual children, with an exploration bonus for nodes that have fewer direct children.

puct_marginal

Q_branch = subtree_audited / (subtree_gpu_hours + tau)
         + 0.5 * recent_marginal_yield_rate
UCB = Q_branch + same exploration bonus

If the last >=3 direct children produced zero audited successes, Q_branch is halved before adding the recency term.

The numbers in this section are toy inputs for explanation, not additional run results. They mirror the code-level fields used by select_puct_portfolio and select_puct_marginal_portfolio.

Why Verifier can be called twice

Verifier is deterministic, so with the same proposal, manifest, budget, and used GPU-hours it returns the same launchability result. It is still used in two places with different responsibilities.

call sitepurposewhat happens if it rejects
After Refiner proposal generationProduces verifier_results[] for the Supervisor prompt. Supervisor sees accepted/rejected, warnings, blocking reasons, and normalized proposals.Supervisor can choose repair or avoid non-launchable proposals.
Immediately before Complexa launchFresh fail-safe and normalized proposal source for backend.render(...).Controller raises before starting Complexa; a non-launchable config cannot reach the executor.

This is defense-in-depth. The first check informs the LLM Supervisor; the second check is the deterministic final launch gate. Verifier has no sampling, no token generation, and no external model call.

Verifier is pure Python, not self-validation by an LLM

The Verifier is implemented as deterministic manifest and budget logic, conceptually:

verify_proposal_against_manifest(proposal, manifest, budget, used_gpu_hours):
    parse proposal schema
    reject unknown config paths
    reject hard-bound violations
    warn on soft-bound violations
    reject forbidden benchmark/evaluator edits
    recompute active_path_multiplier and expanded_work_product
    reject expected_gpu_minutes or remaining-budget violations
    require rollback_condition
    return VerificationResult(...)

The Refiner is flexible but can hallucinate paths or unsafe values. The deterministic Verifier cannot invent scientific experiments, but it can hard-check whether a Refiner proposal is safe to run.
