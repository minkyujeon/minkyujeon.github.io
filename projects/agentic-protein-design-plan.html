1. Rigorous Plan Evaluation

Verdict: the current plan is impactful and doable, but only if it is kept narrow at first: budget-aware operator portfolio selection, not full autonomous research, not denoiser finetuning, and not affinity-predictor-heavy optimization from the start.

DimensionAssessmentReasoning
Scientific impact High The project moves beyond static TTO config search toward target-conditional compute allocation, operator selection, and trace-based improvement. That is a stronger methods contribution than reporting another sweep.
Feasibility Moderate to good Additive search variants, diagnostic sidecar, and rule-based routing already exist locally. The immediate next steps are engineering-heavy but not blocked on pretraining or new wet-lab data.
Main technical risk Moderate Exp 25/26 cracked 3/5 zero-hit targets via MCTS+G&H (SC2RBD 62.5%, TNFα 50%, H1 12.5%). Risk reduced from “High”. Remaining risk: CbAgo and IL17A are still 0% — the Meta-Harness agent (Phase 3) targets these.
Benchmark quality Good if tiered A single 10 GPU-hour benchmark is too strict. A 10 / 30 / 100 GPU-hour tiered benchmark is much more defensible and better aligned with both our results and the Complexa paper.
Overclaim risk Manageable The project should be framed as a budget-aware portfolio controller and trace-improving harness, not as strict causal inference or a fully general autonomous scientist.

What Makes the Plan Impactful

  • It is not standard Bayesian HPO. The decision problem is sequential, target-conditioned, and stateful, so algorithm selection and budget allocation are a better framing than static hyperparameter tuning.
  • It is grounded by strong negative results. Exp 20 and Exp 21 already ruled out the idea that stochastic or diverse beam alone is enough on the zero-hit frontier.
  • It supports a paper-quality claim. If the controller improves success-vs-budget under matched compute, that is a clear methods contribution even before any new model pretraining.

What Must Be Corrected or Deferred

  • 10 GPU-hours is a triage benchmark, not the full success target. Our own results already justify this correction.
  • Search alone is not enough. The next action set must include reward-shaped MCTS and cleaner local refinement, not only beam-family variants.
  • Absolute affinity prediction should stay late. PPB-Affinity is useful later for reranking, but not as the first-line online controller signal.
  • External model-family selection should come after the intra-Complexa portfolio is stable. BindCraft and other proposers are valuable later portfolio arms, not the first engineering step.

Go / no-go criterion: the project remains high-value if we can show that a controller with a stronger action set improves first-hit probability or success-vs-budget on the hard-target set at 30 GPU-hours. If even reward-shaped MCTS plus local refinement fails to shift the curve, the story should pivot from “agent beats fixed search” to “hard-target failure analysis and operator limits.”

Phase 1: Explore the Unexplored Action Space

Exp 1 FK Steering Temperature Sweep

Hypothesis: FK Steering with tuned temperature outperforms both Beam Search (hard top-k) and the paper's default FK Steering (β=10) by finding a target-specific explore-exploit balance.

Motivation: In Figure 7, FK Steering with default T=0.1 already ranks 2nd on hard targets. Lower T should improve exploitation (matching Beam Search); higher T should improve diversity on easy targets.

Setup:

  • 3 targets spanning difficulty: IFNAR2 (easy), Claudin1 (medium), CD45 (hard)
  • 5 temperatures: T=0.01, 0.05, 0.1, 0.5, 2.0
  • Fixed: beam_width=8, n_branch=4, deep checkpoints [0,65,130,200,270,340,400]
  • Baseline: Beam Search deep config (same params but hard top-k selection)
  • Total: 15 FK Steering jobs + 3 beam search baselines = 18 jobs

Hydra overrides:

complexa design configs/search_binder_local_pipeline.yaml \
  ++generation.search.algorithm=fk-steering \
  ++generation.search.fk_steering.temperature=0.05 \
  ++generation.search.fk_steering.beam_width=8 \
  ++generation.search.fk_steering.n_branch=4 \
  ++generation.search.step_checkpoints="[0,65,130,200,270,340,400]"

Success criteria: Does any T value beat both (a) Beam Search and (b) default FK Steering (T=0.1) on unique successes for at least one difficulty tier?

Expected outcome: T=0.01-0.05 should approach Beam Search performance on hard targets. T=0.5-2.0 should approach Best-of-N diversity on easy targets. If T=0.1 is already optimal across all targets, this tells us the default was well-chosen and the value of adaptive T is limited.

Cost: ~18 GPU-hours (1h per job on H100)

Exp 2 Denoising Temperature Sweep

Hypothesis: sc_scale_noise controls trajectory diversity independently from the search algorithm. Higher temperature produces more structurally diverse initial trajectories, and when combined with beam search pruning, may yield more diverse final successes.

Motivation: The companion paper (Latent Generative Search, Table 2, Sec B.4) tested 4 aggregate sampling noise combinations across 75 solvable targets from the 127-target panel and found (bb=0.1, lat=0.4) solves the most unique targets (36 vs 33 for default). But these were aggregate hit counts — they did not break down which noise level helps hard targets specifically, or reveal the self-pass vs. MPNN-pass divergence. We test with per-target pass rate analysis on hard targets (CD45) and easy targets (PDL1).

Setup:

  • 2 targets: PDL1 (easy, 100% self pass), CD45 (hard, 25% self pass)
  • 4 noise scales: 0.05, 0.1 (default), 0.2, 0.4
  • Applied to both backbone and sequence: bb_ca.sc_scale_noise and local_latents.sc_scale_noise
  • Fixed: wide config (beam_width=16, n_branch=4)
  • Total: 8 jobs

Hydra overrides:

complexa design configs/search_binder_local_pipeline.yaml \
  ++generation.model.bb_ca.simulation_step_params.sc_scale_noise=0.2 \
  ++generation.model.local_latents.simulation_step_params.sc_scale_noise=0.2

Metric: FoldSeek cluster count among successes AND total unique successes. We want to distinguish "more diverse but fewer successes" from "more diverse AND more successes."

Risk: Too high noise may break the model's learned priors, producing clash-ridden or unphysical structures. 0.4 may already be too high.

Cost: ~8 GPU-hours

Exp 3 Composite Reward Signals

Hypothesis: Adding secondary reward signals to beam search pruning improves the quality of surviving beams, particularly for hard targets where iPAE-only pruning may discard structurally viable candidates.

Motivation: Current TTO uses only AF2 iPAE for pruning. The codebase has 4 reward models: AF2 (17 metrics), RF3 (14 metrics), TMOL (H-bonds, electrostatics), and Bioinformatics (shape complementarity, dSASA). TMOL and Bioinformatics are fast (no neural network inference) and could add signal at negligible cost.

Setup:

  • 2 hard targets: CD45 (25% deep pass rate), BetV1 (18.8% deep pass rate)
  • 3 reward compositions:
    (a) iPAE only (baseline, weight -1.0)
    (b) iPAE (-1.0) + pLDDT (0.3) + i_ptm (-0.2) [multi-AF2-metric]
    (c) iPAE (-1.0) + i_ptm (-0.2) + tmol H-bond (0.1) [cross-model composite]
  • Fixed: deep config (beam_width=8, n_branch=3)
  • Total: 6 jobs

Hydra overrides (option b):

complexa design configs/search_binder_local_pipeline.yaml \
  ++generation.reward_model.reward_models.af2folding.reward_weights.plddt=0.3 \
  ++generation.reward_model.reward_models.af2folding.reward_weights.i_ptm=-0.2

Success criteria: Does any composite reward achieve higher pass rate than iPAE-only on hard targets?

Risk: Additional reward signals may conflict with iPAE, causing the beam to optimize for the wrong objective. Need to verify TMOL/bioinformatics actually have non-zero variance on Proteina-generated backbones (prior work found clash and RoG had zero variance).

Cost: ~6 GPU-hours

Exp 4 Sequence Refinement for High-Backbone / Low-Sequence Targets

Hypothesis: For targets where the model generates good backbones but weak sequences (self_pass << mpnn_pass or backbone iPAE is good but self scRMSD is poor), AF2-guided sequence hallucination can rescue designs.

Motivation: HER2_AAV-deep shows self 3.1% but mpnn 31.3% — the backbones are viable but the co-designed sequences are suboptimal. The codebase has a sequence_hallucination refinement stage that optimizes sequences via AF2 gradients.

Important context: In Figure 7, "Generate & Hallucinate" (which uses this refinement) performed worst among all 5 algorithms. However, that applied hallucination to all designs. Our hypothesis is that applying it selectively — only to designs with good backbones but poor sequences — could be beneficial.

Setup:

  • Take top-16 designs from HER2_AAV-deep (clear backbone > sequence gap)
  • Also top-16 from PDL1-deep (100% self, 37.5% mpnn — moderate gap)
  • Apply sequence hallucination: 45 softmax iters + 15 greedy iters with default loss weights
  • Re-evaluate with AF2 refolding

Hydra overrides:

complexa design configs/search_binder_local_pipeline.yaml \
  ++generation.refinement.algorithm=sequence_hallucination \
  ++generation.refinement.enable_soft_optimization=true \
  ++generation.refinement.enable_greedy_optimization=true

Success criteria: Does refinement increase self pass rate on HER2_AAV from 3.1% toward mpnn's 31.3%?

Cost: ~4 GPU-hours

Exp 7 Classifier-Free Guidance (CFG) Scale Sweep

Hypothesis: Increasing guidance_w above 1.0 amplifies the conditional signal (target-conditioned velocity field minus unconditional prior), pushing the denoising trajectory harder toward the target binding interface. This is analogous to high CFG scales in text-to-image diffusion models and could dramatically improve backbone quality for hard targets.

Motivation: This is the single most impactful unexplored knob. The default guidance_w=1.0 means CFG is completely disabled — v_guided = 1.0 * v_cond + 0.0 * v_uncond = v_cond. With guidance_w > 1.0, the formula becomes v_guided = w * v_cond + (1-w) * v_uncond, which subtracts the unconditional prior and amplifies the target-specific signal. In image generation, CFG scales of 2–7 are standard and transformative for quality. No prior Proteina experiment has tested this.

Setup:

  • 2 targets: CD45 (hard, 25% deep baseline), BetV1 (hard, 18.8% deep baseline)
  • 4 guidance weights: 1.0 (control), 1.5, 2.0, 3.0
  • Fixed: deep config (beam_width=8, n_branch=3)
  • Total: 8 jobs (4 weights × 2 targets)

Hydra overrides:

complexa design configs/search_binder_local_pipeline.yaml \
  ++generation.args.guidance_w=2.0

Risk: Very high guidance_w (3.0+) may cause mode collapse or degenerate structures, similar to how high CFG in image models produces oversaturated images. Also requires the model to have been trained with unconditional dropout — need to verify.

Success criteria: Does guidance_w > 1.0 increase self pass rate on CD45 above the 25% beam baseline?

Cost: ~8 GPU-hours

Phase 1.5: Improve Generation Quality (Engineering Required)

These experiments go beyond tuning existing hyperparameters — they modify the model or the inference procedure itself. Higher engineering cost, but potentially the highest ceiling for hard targets.

Exp 10 LoRA Finetuning on Successful Binders

Hypothesis: The Complexa denoiser was pre-trained on general protein structures. Finetuning with LoRA adapters on successful binder designs shifts the model’s prior distribution toward structures that actually bind, improving backbone quality at generation time.

Motivation: For CD45, only 25% of beam-selected backbones pass even with deep search. The model’s prior simply doesn't cover the right conformational space for hard interfaces. Rather than searching harder in a bad prior (Exp 1–7), we shift the prior itself.

Two approaches:

  • Option A — Supervised: Finetune on all ~500+ successful designs across all 11 targets. The model learns “what successful binders look like” in general. Risk: dominated by easy targets (IFNAR2 has 100% pass rate, contributing most training data).
  • Option B — DPO-style: Train on (successful backbone, failed backbone) pairs from the same target. The model learns to distinguish what makes a binder succeed vs. fail for a given interface type. More data-efficient for hard targets (CD45: 8 positive + 24 negative pairs from deep). Conceptually analogous to RLHF/DPO for LLMs — aligning the model toward reward-preferred outputs.

Setup:

  • Collect all successful/failed backbone structures from Phase 0 + Phase 1 runs
  • Add LoRA adapters (rank 16–32) to the Complexa denoiser transformer blocks
  • Train for 500–2000 steps with small learning rate (1e-5)
  • Evaluate: run deep beam search on CD45 and BetV1 with finetuned model

Engineering required: LoRA wrapper around Complexa denoiser, denoising score matching loss on successful structures (Option A) or DPO preference loss (Option B).

Risk: Overfitting — only ~500 total successes across all targets, and only 8 for CD45. Mitigation: low LoRA rank, early stopping on held-out target, regularization to base weights.

Cost: ~20 GPU-hours training + ~10 GPU-hours evaluation. ~1-2 days engineering.

Exp 11 Trajectory Recycling / Warm-Starting

Hypothesis: Successful designs follow “good paths” through the denoising trajectory. By saving intermediate latent states from successful trajectories and re-starting new beam searches from those states (with small noise perturbations), we can do local search in trajectory space to find more successes without re-exploring the global fold topology.

Motivation: The Complexa denoiser runs 400 steps. The first ~130 steps determine the global backbone fold topology. For CD45's 8 successful designs, steps 0–130 found the right fold geometry. By seeding new runs from t=130 of these trajectories + noise, we skip the hardest part (finding the right fold) and explore variations of the interface geometry.

Setup:

  • Phase A: Modify beam search to save intermediate latent states at each checkpoint
  • Phase B: Run deep beam search on CD45, collect latent states at t=130 from the 8 successful trajectories
  • Phase C: Start 8 new beam searches, each seeded from one successful t=130 state + Gaussian noise (σ=0.05–0.2)
  • Compare: unique successes from warm-started runs vs. cold-started runs with same GPU budget

Engineering required: Modify search/beam_search.py to save/load intermediate trajectory states. Add noise perturbation to loaded states. ~1 day engineering.

Risk: If the 8 successful trajectories share a single mode (all converged to the same backbone topology), warm-starting will produce near-duplicates rather than diverse new designs. Check with FoldSeek clustering first.

Cost: ~8 GPU-hours runs + ~1 day engineering

Phase 2: Build the Agent

Exp 5 Diagnostic Signal Extraction

Hypothesis: A cheap single-pass diagnostic (N=32, ~5 min generation + ~20 min AF2 eval) contains enough signal to predict target difficulty and route to the optimal algorithm/config.

Setup:

  • For all 11 targets, extract from notto runs (N=200): mean iPAE, iPAE variance, % below 0.226 threshold, iPAE distribution skewness, mean pLDDT, mean RMSD, sequence entropy
  • Correlate each signal with: (a) optimal config (deep vs wide), (b) self pass rate, (c) difficulty tier
  • Test whether a simple threshold rule (e.g., "if mean_iPAE > 0.7, use deep") achieves >80% routing accuracy

Expected output: A decision tree mapping diagnostic statistics to recommended config.

Cost: Analysis only — all 11 notto runs complete with evaluation data

Exp 6 LLM Routing Validation

Hypothesis: An LLM reading structured JSON (diagnostic statistics + target metadata) can route targets to configs more accurately than simple threshold rules, especially for medium-difficulty targets where the optimal config is ambiguous.

Setup:

  • Prepare structured JSON per target: iPAE distribution (mean, std, min, max, quartiles), pLDDT distribution, notto pass rate, target metadata (length, hotspot count, interface type)
  • Prompt: "Given these diagnostic statistics, recommend: algorithm (beam-search / fk-steering), beam_width, n_branch, checkpoint schedule, and explain your reasoning"
  • Compare LLM routing vs. threshold rules vs. oracle (actual best config from sweep)
  • Leave-one-out: train on 10 targets, predict 11th

Key question: Does the LLM's ability to reason about biological context (e.g., "this target has a large flat interface, which typically benefits from wider beam search") add value beyond statistical pattern matching?

Cost: API calls only

Exp 8 Multi-Round Adaptive Pipeline

Hypothesis: An agent that runs diagnostic Round 0, selects config for Round 1, reads results and adjusts for Round 2 produces more unique successes per GPU-hour than any fixed strategy.

Setup:

  • 3 targets: IFNAR2 (easy), SpCas9 (medium), CD45 (hard)
  • Agent protocol:
    Round 0: single-pass N=32 diagnostic (~0.5 GPU-hr)
    Round 1: agent-selected config based on diagnostic (~1 GPU-hr)
    Round 2: agent adjusts based on Round 1 results (~1 GPU-hr)
    Round 3: final run with refined config (~1 GPU-hr)
  • Compare total unique successes after 3.5 GPU-hours vs:
    (a) fixed deep config x3 rounds = 3 GPU-hours
    (b) fixed wide config x3 rounds = 3 GPU-hours
    (c) oracle (always use best config) x3 rounds = 3 GPU-hours

Agent decisions between rounds: Adjust beam_width, n_branch, FK temperature, nsamples, sc_scale_noise, guidance_w, reward composition. Record all decisions and reasoning for analysis.

Success criteria: Agent achieves ≥80% of oracle performance on at least 2/3 targets.

Cost: ~15 GPU-hours (3.5h x 3 targets + baselines)

4. Phase 2 Vision: From Diagnostic Routing to State-Aware Inference

Live-plan boundary: from this section onward, the page switches from preserved historical planning context to the current working architecture and benchmark agenda. If you only read one part of the plan page, this is the right starting point.

Phase 1 Foundation: Diagnostic Routing Already Works

What We Have Built (Not Generic Hyperparameter Search)

Phase 1 established that inference-time control in Proteina-Complexa is already target-aware, not a blind knob sweep. The evidence:

  • Target-specific bottleneck identification from a single diagnostic round (N=32, ~6 GPU-hours).
  • A validated routing tree (3/4 correct) that reads Self/MPNN gap + per-criterion pass rates to select the right intervention.
  • Non-additive intervention effects: noise + hallucination is worse than either alone (p=0.032).
  • Intervention specificity: hallucination helps only targets with a backbone>sequence gap; noise=0.20 helps only targets with an iPAE bottleneck. Both destroy the other type of target.
  • 100+ experiments across 11 targets, 3 configs, and 15+ intervention conditions, all cross-validated at the per-design level.

This is the foundation. The next step is not to replace this with something else, but to extend it — from single-round routing into multi-round, state-aware inference with broader target coverage.

Phase 2: From Single-Round Routing to Multi-Round Adaptive Inference

The current routing tree handles Round 1 → Round 2 (diagnose → intervene). The next stage extends this to a full adaptive loop:

  • Multi-round feedback: After applying an intervention, read the new diagnostic signals and adjust. For example, if a target remains interface-limited after one round, allocate the next round to more structural exploration; if sequence quality becomes the bottleneck, switch to sequence-focused refinement.
  • Broader target validation: Currently validated beyond the original 4 targets. Exp 15a-d completed for SpCas9 and CrSAS6, Exp 16a-i completed diagnosis for BHRF1/SC2RBD/VEGFA, Exp 17a-x covered the remaining packaged targets, and Exp 20 then tested beam, stochastic beam, and diverse beam on the zero-hit frontier with all runs staying at 0/16 self and 0/16 MPNN. Later Exp 25/26 changed that picture: SC2RBD, TNFα, and H1 now have non-zero completed runs under MCTS + local refinement, while CbAgo remains 0/0 and IL17A remains 0 self.
  • Scaling curve as the key figure: Compare unique successes vs GPU-hours for the adaptive agent against fixed Beam Search, fixed Best-of-N, and fixed hallucination baselines.

Failure Decomposition → Intervention Selection → Improvement Trace Accumulation

The next research step is not to throw away the current routing work. It is to make it more agentic by replacing a single scalar interpretation of failure with an intervention-grounded diagnostic layer.

In the current system, test-time optimization is still dominated by structural confidence proxies such as interface confidence and backbone consistency. Those signals are useful, but they are not the same thing as binding thermodynamics. A structure predictor can believe a pose is plausible while the interface is still poorly packed, electrostatically mismatched, or thermodynamically weak.

Our Phase 2 controller therefore shifts from “maximize one reward” to “diagnose why this target is failing, choose the next intervention, and remember which intervention actually improved it”. This is closer to actionable counterfactual feedback than to a strict causal model, but it is a better fit for a fixed generator plus inference-time search.

Why PPB-Affinity Is Deferred

Absolute affinity modeling is still important, but it is not the right first benchmark for the full agentic design system. We will keep PPB-Affinity as a later auxiliary benchmark for an absolute ΔGbind reranker rather than use it as the primary de novo design benchmark.

The primary benchmark for the agent remains the actual hard design targets (SC2RBD, CbAgo, H1, IL17A, TNFα), where the question is whether diagnostic-driven search and targeted intervention unlock designs that the current fixed search stack misses.

Why this is a distinct contribution: Meta-Harness optimizes a single harness to be the best. Portfolio composition instead asks: which combination of harnesses covers the most distinct modes? This is the protein-design analogue of ensemble diversity in ML — ensembles of diverse weak learners outperform a single strong learner.

Metric: FoldSeek structural clustering at TM-score thresholds 0.5 and 0.7. Report both total unique successes and cluster count. The portfolio wins if cluster_count(portfolio) > cluster_count(fixed) even when total_successes(portfolio) ≤ total_successes(fixed).

Cost: Included in the future scaling-benchmark compute budget. Status: Future

Priority 6 Intermediate Backbone Diagnostic Pipeline

Motivation: Current diagnostics only see the final structure. But the MCTS tree stores intermediate backbone structures at every checkpoint (MCTSState.x_t and x_1_pred in proteinfoundation/utils/mcts_utils.py). A design can have a good backbone at t=200 that gets corrupted by t=400 — or a good interface at t=150 that the reward function prunes away. Without intermediate diagnostics, we cannot distinguish “the model never generates good intermediates” from “the model generates good intermediates but the search destroys them.” This is the key enabling technology for both failure analysis (Priority 7) and Meta-Harness-style agent optimization (Phase 3).

What MCTSState already stores (verified from source):

Signals to extract at each checkpoint:

SignalWhenWhat It RevealsCost
Backbone Cα contacts to target hotspots Every checkpoint Is the binder approaching the correct binding site? How many interface contacts form and when? A trajectory where contacts form at t=200 then drop at t=300 indicates reward misguiding. Free (geometric computation on x_1_pred coordinates)
Interface contact count t=100+ (after rough fold) Number of binder-target Cα pairs within 8Å. Monotonic increase = good trajectory. Decrease = the search is abandoning a promising interface. Free
Backbone radius of gyration Every checkpoint Is the binder forming a compact fold or staying extended? Compact early = the topology is locking in. Useful for detecting “the binder never folds” failure mode. Free
ESMFold pLDDT (fast proxy) Every N checkpoints (e.g., t=0, 100, 200, 300, 400) Quick structural confidence without full AF2 Multimer. Identifies whether the sequence is confident about the intermediate backbone — separates backbone quality from sequence quality. Cheap (~2s per structure via ESMFold)
Self vs MPNN designability at intermediate backbone 1-2 key checkpoints (e.g., t=200, t=400) If MPNN can design a good sequence for the t=200 backbone but not the t=400 backbone, the late denoising steps are corrupting a designable structure. This is the strongest signal for “backbone was good, late refinement broke it.” Expensive (MPNN + AF2 refold; use sparingly)
Reward trajectory Every checkpoint The reward value at each MCTS decision point. Already computed by the MCTS search (stored in cumulative_reward). Monotonic increase = reward guides correctly. Spike-then-drop = reward is misleading the search. Free (already computed)

Priority 7 Meta-Harness Agent: LLM-Driven Harness Optimization

Revised vision (April 10): The rule-based router (v1) selects from a fixed menu of 5 operators. This is sufficient for known patterns but cannot discover novel configurations. The Meta-Harness pattern goes further: an LLM agent reads full execution traces (intermediate backbone structures, per-checkpoint rewards, structural quality trajectories) and proposes structural modifications to the inference pipeline. It doesn’t select from a menu — it writes new configurations by reasoning about why the previous one failed at the trajectory level.

What makes this different from rule-based routing:

CapabilityRule-Based Router (Phase 2)Meta-Harness Agent (Phase 3)
Action space Fixed menu: {default, noise, halluc, MCTS+G&H, MCTS+G&H+pLDDT} Open-ended: any valid Hydra override combination, including novel reward functions, checkpoint schedules, and refinement parameters
Input signal 5 scalars: self_rate, mpnn_rate, iPAE_pass%, pLDDT_pass%, gap Full trajectory diagnostic: per-checkpoint backbone structures, contact trajectories, reward curves, ESMFold pLDDT at intermediates, final evaluation + all prior traces
Failure reasoning “self_rate=0 → zero-hit → try MCTS+G&H” “Interface contacts form at t=200 (12 contacts) but drop to 5 by t=300. The reward function prunes good intermediates during late refinement. Proposed fix: add contact_preservation=0.3 to reward after t=200, reduce exploration_prob from 0.3 to 0.1 in late checkpoints.”
Discovery potential None — can only select what we already know works Can discover interventions we haven’t manually tested (e.g., checkpoint-specific reward weights, novel noise schedules)

What This Project Is and Is Not

It is: a two-phase system for de novo binder design that mirrors the Meta-Harness architecture: a frozen generative model (Complexa) surrounded by an optimizable inference-time harness (search algorithm, reward function, refinement schedule). Phase 2 uses a rule-based router to select from validated operators. Phase 3 uses an LLM agent that reads intermediate backbone trajectory diagnostics and proposes novel harness configurations — going beyond menu selection to open-ended pipeline optimization.
It is not: a generic hyperparameter sweep, a static best-config search, or a simple rule-based router (though rules are the starting point). The full system aims to show that an agent-discovered configuration outperforms any manually designed one by reasoning about why the pipeline fails at the trajectory level.
Working thesis (confirmed by Exp 25/26): the model covers the correct fold space — H1 yields hits in 19.4h (paper: 86.3h/hit), TNFα in 16.2h (paper: 31.7h/hit), SC2RBD is cracked at 62.5% (paper: 0%). Phase 2 proves this via rule-based routing. Phase 3 proves that intermediate backbone diagnostics + LLM reasoning can discover configurations that no hand-designed rule would find — the Meta-Harness thesis applied to protein design.
