1. Why This Companion Page Exists

The main results page is now an audited empirical map. The plan page is the forward-looking research thesis. This page sits in between them. It is for the messy but important middle layer: implementation decisions, boundary definitions, LLM-backend experiments, and the gap between the original idealized diagram and the current code.

Page contract. This page is not the benchmark scoreboard and not the final thesis. It is a technical companion for understanding how the current system is wired, what evidence exists, and where the project is still provisional.

3. What autoresearch_v2 Actually Is

The current implementation is best described as PUCT tree search over structured JSON programs. The evolvable object is InferenceProgram, not arbitrary Python source. Each node stores target, stage config, lineage metadata, and optionally a gated code patch. The program is content-hashed so semantically identical nodes deduplicate even if they were reached through different histories.

Central object
InferenceProgram JSON/dataclass with semantic content hash.
Search loop
Global PUCT selection over already-scored nodes, no rollout phase.
Executor
Compiles program fields to Hydra overrides and relaunches the fixed Complexa generation stack.
Evaluator
Reads rewards_*.csv for the raw online proxy and upgrades to FoldSeek-unique successful self binders when audited evaluation_results/ artifacts exist.
LLM mutation surface
One or two targeted field edits per step, under strict allowlist and range validation. The LLM is a local mutator, not the tree policy.

What Is Fixed

  • Backbone model weights and the main Proteina-Complexa runtime.
  • The online evaluator definition for a given v2 version.
  • The Tier D boundary: no uncontrolled edits to AF2 internals or backbone code.

What Is Mutable

  • Search policy: beam, best-of-n, fk-steering, MCTS-related knobs.
  • Sampling policy: steps, guidance, recycle count, selected schedule overrides.
  • Refinement and filtering policy, and eventually some controlled multi-stage behavior.

5. What the LLM Does and Does Not Do

In the current v2 implementation, the LLM is not a free-form code-writing scientist in the hot loop. Its primary operational role is much narrower and more controlled.

Role What it does now What it does not yet do
LLMMutator Reads parent program, compact feedback, optional ideas, then emits a tool call with 1-2 field updates. Does not freely rewrite arbitrary source code every step.
IdeaResearcher Runs a multi-turn tool-calling loop to produce target-specific hypothesis notes and idea-bank entries. Is not the main executor or evaluator.
Tree search driver Owns node selection, mutation attempts, execution scheduling, and score backup. Does not delegate the control loop itself to the LLM.

Practical summary. Today the LLM is closer to a config mutation operator than a fully general code-generation agent. The plan page pushes toward a larger software-evolution framing, but the current implementation remains intentionally narrower so the search space stays validatable and the run store remains interpretable.

7. Benchmark Contract and Why It Is Still Under Revision

The biggest unresolved issue is not whether the loop runs. It is whether the online score used by the tree is exactly the same object as the score used by the audited results page. Right now those two things are related, but not identical.

Current online proxy

The tree currently optimizes an online score computed directly from the design-run reward CSVs. In the current code this already uses the paper-aligned af2folding_i_pae < 7/31 and af2folding_plddt_log > 0.90 thresholds, which makes the online proxy much closer to the audited heuristic criterion than the earlier stricter 0.18 proxy.

Paper-style audited metric

The audited results page still cares about post-hoc outcomes after redesign, refolding, and additional structural checks such as self-consistency RMSD and structural uniqueness. Those remain slower, external analyses and still cannot simply replace the online reward in the hot loop.

Current risk. “The loop runs” and “the benchmark is apples-to-apples” are two different statements. The first is supported by the current runtime evidence. The second still depends on completing the same audited evaluation contract used by the preserved heuristic pages.

9. Design Implications

The guiding goal is not to make the architecture more baroque. It is to make the benchmark honest. That means expanding the comparator library beyond deep / wide, keeping the current pipeline stable, and tightening the bridge between online tree-search rewards and the audited post-hoc results.

Practical implication. Keep autoresearch_v2 structurally simple, keep local LLM backends working through SLURM-served OpenAI-compatible endpoints, and use this page as the place to document each narrowing step from broad research idea to benchmarkable implementation.
