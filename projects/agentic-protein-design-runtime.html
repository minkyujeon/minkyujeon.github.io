1. What Is Already Verified

Several layers are already closed: the top-level workspace migration, the self-hosted LLM serving path, the real Complexa generation path, and the local PUCT + mutator loop. The remaining open issue is not whether the pipeline runs, but whether the preserved post-hoc evaluation stack is producing the exact audited artifacts required for a fair website-style comparison.

Interpretation rule. The current uncertainty is not “does the pipeline run?” It does. The remaining uncertainty is whether the exact evaluation_results/ contract used by the website and preserved heuristic runs is being reproduced end-to-end.

3. Tree Search vs. LLM: Separate Roles

The runtime works best when these two pieces are kept conceptually separate.

Component What It Decides What It Does Not Do
PUCT tree search Selects which parent program to expand next, tracks visits, and allocates search budget across promising branches. Does not author hypotheses or free-form code. It never writes the child itself.
LLM mutator Given one parent and limited feedback, proposes one or two local field edits to produce a child program. Does not see the whole tree, sibling branches, or global search state. It is not the outer controller.
Complexa runtime Actually generates, folds, redesigns, and scores candidate binders. Does not know about tree search or benchmark fairness.

Short version. Tree search is the outer policy. The LLM is the local proposal operator. Complexa is the execution engine.

4. Easy Targets: What the Loop Actually Does

On easy targets such as 02_PDL1, the seed program is already strong. The runtime therefore behaves less like “discovering an algorithm from scratch” and more like “testing local variations around an already successful recipe.”

Observed Behavior

  • The balanced seed already scores well on the online proxy.
  • Baseline deep can reach the expected 32/32 online success pattern.
  • LLM-guided runs often begin from a still-good but slightly weaker seed than deep.
  • When a child fails here, it is often a runtime or memory problem rather than a pure “no good designs exist” signal.

What Tree Search Is Really Testing

  • Do small local edits improve a seed that is already good?
  • Do they improve the same online metric as the seed, or just create instability?
  • Can a child stay inside the single-GPU memory envelope?

Why this matters. On easy targets, the main risk is not zero-hit failure. It is generating children that are worse than the seed or fail at runtime despite coming from a plausible local hypothesis.

5. Hard Targets: What Changes

On hard targets the seed is no longer a generic balanced beam configuration. The intended contract is to begin from a routed heuristic seed such as MCTS + refinement or reward-shaped MCTS, and then let the tree explore local edits around that family. 

hard target
→ routed seed profile
→ seed executes real Complexa
→ tree treats this heuristic as the local starting point
→ LLM proposes one bounded edit
→ child is evaluated
→ tree decides whether that branch earns more budget

Why Hard Targets Are Different

  • The interesting question is not whether the default beam baseline works. It usually does not.
  • The interesting question is whether the tree can improve or stabilize a nontrivial heuristic family such as routed MCTS + hallucination.
  • The local mutator is still small-scope. It is not inventing a brand-new search family in one step.
  • The routed seed itself is part of the benchmark contract, so seed-profile validity matters just as much as LLM proposal quality.

6. Qwen vs. Gemma-4 in Practice

Model What Worked Main Failure Mode Observed So Far
Qwen3.6 Most stable self-hosted backend so far. Tool-calling and basic mutation proposals are reliable enough to keep using for runtime verification. Still subject to the same downstream runtime and apples-to-apples analysis issues as the rest of the pipeline.
Gemma-4 Serving and mutation loop both work. Observed a concrete child failure where a valid proposal still led to an OOM during real Complexa execution.

Operational choice. The current verification pass therefore focuses on Qwen for hard-target reruns, not because Gemma-4 never works, but because Qwen is behaving as the more reliable local mutator backend right now.

7. Runtime Lessons Worth Keeping

Some of the recent trial-and-error is not just disposable debugging. A few failures exposed structural contracts that are worth preserving as part of the project memory even after the immediate bug is fixed.

Lesson What Happened Why It Matters
SLURM batch cannot trust BASH_SOURCE[0] Path discovery can collapse into /var/spool/... instead of the repo root when a batch script is launched from SLURM. Runtime reproducibility depends on explicit repo resolution. This is now part of the launcher contract, not a cosmetic cleanup.
Historical heuristic labels need translation Names like msg5, rws5, and paper_gh_1pct encode percentages in human shorthand, but the v2 schema expects normalized floats such as 0.05 or 0.01. Porting preserved heuristics into v2 is not a copy-paste task. The semantic contract has to be translated carefully.
A valid child can still be a runtime failure One Gemma-4 child passed validation and tree insertion, but later OOMed during real Complexa execution. Validation and execution safety are different layers. Easy-target search can still fail because of runtime geometry, not because the idea was irrational.
Post-hoc analysis must inherit the same Python environment The first attached analysis attempts failed because helper subprocesses fell back to the wrong Python and broke SolubleMPNN. Apples-to-apples comparison depends on environment continuity, not only on calling the right command name.

Practical takeaway. The remaining work is not just “prompt the LLM better.” It is making the runtime contract explicit enough that the same tree, mutator, and evaluation stack mean the same thing across every SLURM job.

8. Why Apples-to-Apples Is Still a Runtime Issue

The online score is easy to compute. The website-style audited score is not. The latter depends on the preserved evaluation_results/ tree, redesign outputs, and FoldSeek uniqueness analysis.

Online score
Already available from rewards_*.csv. Used immediately by the tree.
Post-hoc analysis
Attached to the executor and used to materialize audited benchmark artifacts.
Audited binder artifact
The binder audit relies on sequence_type_stats.json files together with FoldSeek clustering outputs.
Unique-success tree score
Tree ranking uses unique successes when audited artifacts exist and otherwise falls back to the online score.

Interpretation rule. Comparisons between tree search and heuristics should be read under the same audited binder-analysis contract used for the preserved benchmark pages.

9. Audit Checklist

  1. Use sequence_type_stats.json and FoldSeek clustering outputs under evaluation_results/ as the audited benchmark artifacts.
  2. Read the corresponding paper_audited_metric entries in the v2 summary files.
  3. Compare baseline and tree runs only under the same audited contract.
  4. Use routed hard-target comparisons only against the matching preserved heuristic families.

Practical state. The runtime page is a reference for how the same code path produces the audited outputs needed to compare tree search against preserved heuristic baselines.
