3. Tree Search vs. LLM: Separate Roles

The table below lists the current responsibilities of each component.

Component What It Decides What It Does Not Do
PUCT tree search Selects which parent program to expand next, tracks visits, and allocates search budget across promising branches. Does not author hypotheses or free-form code. It never writes the child itself.
LLM mutator Given one parent and limited feedback, it can inspect Complexa read-only, then proposes one to four bounded updates to produce a child program. It can also carry forward ideas, attach hypothesis fields, and append or condition later stages. Does not see the whole tree, sibling branches, or global search state. It is not the outer controller, and it does not patch arbitrary source code in the hot loop.
Complexa runtime Actually generates, folds, redesigns, and scores candidate binders. Does not know about tree search or benchmark fairness.

Role split. Tree search selects parents and backs up scores. The LLM mutator proposes child programs. Proteina-Complexa executes the generated overrides.

4. Easy Targets: What the Loop Actually Does

This section records the observed runtime pattern on easy targets such as 02_PDL1.

Observed Behavior

  • The balanced seed scores well on the online proxy in the recorded easy-target runs on this page.
  • The deep baseline reaches 32/32 online success on the recorded PDL1 run on this page.
  • Child runs on these targets include both successful children and failed children.
  • Recorded child failures include runtime and memory failures.

Observed Tree Behavior

  • The tree evaluates local edits around the stored seed program.
  • The stored child programs on these runs include improvements, ties, and regressions relative to the seed.
  • The stored run records include runtime failures as well as scored children.

Runtime note. On easy targets, child failures include both metric regressions and runtime failures.

5. Hard Targets: What Changes

Current hard-target runs use two settings: seeded runs begin from a routed heuristic seed, and discovery runs begin from a smaller neutral seed.

hard target
→ choose a routed or neutral seed profile
→ seed executes real Complexa
→ tree treats that seed as the current starting point
→ LLM proposes bounded edits within the allowed control surface
→ child is evaluated
→ tree decides whether that branch earns more budget

Current Hard-Target Contract

  • Seeded runs start from a routed profile such as hallucination, reward-shaped search, or another preserved profile.
  • Discovery runs start from a neutral profile and use the same bounded mutation surface.
  • The mutator stays inside the schema and validation layer and does not execute arbitrary source patches in the hot loop.
  • In both settings, the tree allocates additional budget only after the child is scored.

7. Runtime Lessons Worth Keeping

This section records runtime conditions that changed execution behavior in the logged runs.

Lesson What Happened Why It Matters
SLURM batch cannot trust BASH_SOURCE[0] Path discovery can collapse into /var/spool/... instead of the repo root when a batch script is launched from SLURM. The launcher now resolves the repo root explicitly.
Historical heuristic labels need translation Names like msg5, rws5, and paper_gh_1pct encode percentages in human shorthand, and the v2 schema must preserve the same semantic units that Proteina-Complexa expects. Preserved heuristics require unit-consistent translation into the v2 schema.
A valid child can still be a runtime failure One Gemma-4 child passed validation and tree insertion, but later OOMed during real Complexa execution. Validation and execution safety are separate layers.
Post-hoc analysis must inherit the same Python environment The first attached analysis attempts failed because helper subprocesses fell back to the wrong Python and broke SolubleMPNN. The attached analysis step requires the same Python environment as generation.

Runtime takeaway. The remaining work is making the runtime contract explicit enough that the same tree, mutator, and evaluation stack produce the same artifact structure across SLURM jobs.

8. Why Apples-to-Apples Is Still a Runtime Issue

The online score is easy to compute. The website-style audited score is not. The latter depends on the preserved evaluation_results/ tree, redesign outputs, and FoldSeek uniqueness analysis.

Online score
Already available from rewards_*.csv. Used immediately by the tree.
Post-hoc analysis
Attached to the executor and used to materialize audited benchmark artifacts.
Audited binder artifact
The binder audit relies on sequence_type_stats.json files together with FoldSeek clustering outputs.
Unique-success tree score
Tree ranking uses unique successes when audited artifacts exist and otherwise falls back to the online score.

Interpretation rule. Comparisons between tree search and heuristics should be read under the same audited binder-analysis contract used for the preserved benchmark pages.
