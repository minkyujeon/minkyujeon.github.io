1. Two Numbers, Two Questions

Every Complexa evaluation produces two different quality signals from the same batch of finals. This page uses both, and keeps them separate throughout.

QuantityFormulaWhat it measures
Pass rate n_successful / n_finals Of the finals Complexa generated, what fraction cleared the online threshold (af2folding_i_pae < 7/31 AND af2folding_plddt_log > 0.9). Invariant to how many samples you generate. Measures the config.
n_successful (count) count of finals clearing threshold Absolute count of threshold-crossing finals. Scales with the size of the final pool and therefore with the sampling budget, not just with config quality.

Reading rule. "Qwen/CD45 reached 33" means 33 successful finals out of 256, i.e. 12.9 % pass rate. Never "a success rate of 33 %." Both numbers appear in every table below.

PUCT uses only n_successful (the count) to rank nodes during the tree search. A program with more finals at the same pass rate will therefore outrank one that found a higher pass-rate config at smaller scale. This shapes the chains below.

2. Executive Summary

  • On PDL1, both LLM trees eventually reached 100 % pass rate at small scale, and both the human-designed v1 deep baseline also achieved 32 / 32 = 100 % at the same scale in a single 30-minute run. The LLM trees then scaled up n_finals to increase absolute count (Qwen: 248 at 97 %, Gemma: 109 at 85 %), trading pass rate for pool size.
  • On CD45 (seed = 0 successful), the v1 noise020 baseline reached 8 / 32 = 25.0 % in 33 minutes. The LLM trees peaked at 21.1 % (Qwen gen 4, Gemma gen 1). Their best-count nodes reached 33 successes at 256 finals.
  • All refinement-only mutations on CD45 (Qwen gens 1–3, each a ~25-minute eval) kept pass rate at 0 %. Only raising nsamples escaped the zero basin.
  • The tree ranks by count rather than pass rate. In these runs, the best-count nodes also use the largest final pools.

Matched-scale summary. On PDL1, the tree reaches 100 % pass rate after one or two accepted mutations, and the human-designed deep baseline also reaches 100 % at the same small scale. On the current completed CD45 runs, the peak tree pass rate remains below the human noise020 baseline.

On CD45, the human baseline has the highest pass rate on this page. noise020 (beam_width=8, n_branch=3, bb_ca & local_latents sc_scale_noise=0.2) clears 25 % of finals in a single 33-minute run. Neither LLM tree matched that quality at any node in its chain. The highest LLM counts on CD45 occur at 256 finals rather than 32.

What this is NOT. These numbers are the online proxy, computed from raw Complexa finals. The website's audited "After AF2 Refolding + ProteinMPNN Redesign" metric uses a different gate (ProteinMPNN redesign + ESMFold + scRMSD_ca < 1.5 Å + FoldSeek clustering). That post-hoc pipeline has not been run on these outputs yet, so the headline numbers here cannot be compared directly against the audited human baselines.

3. Pipeline and Role Separation (Human vs Tree vs LLM)

The v2 autoresearch loop has three cooperating actors. This section separates which parts are fixed before the run and which parts are chosen during the run.

Scope of this section. The loop below describes the initial campaign recorded on this page. The current local working tree described on the notes and runtime pages now includes a bounded local tree summary in the mutator prompt.

End-to-end loop

   ┌─ seed InferenceProgram (campaign seed profile)   ← SEED
   │
   ▼
   ┌──────────────────────────────────────────────────┐
   │  tree.select()   [PUCT — v2/tree.py]             │   pure tree search
   │  pick next parent node by UCB over               │   (no LLM here)
   │  primary_successes + visits + virtual_loss       │
   └──────────────────────────────────────────────────┘
         │  parent program + target
         ▼
   ┌──────────────────────────────────────────────────┐
   │  mutator.propose()   [LLM — v2/mutator.py]       │   per-expansion LLM call
   │  Input : parent program, Tier-A surface spec,    │   (stateless:
   │          feedback blob, run_history, idea bank   │    no tree topology,
   │  Output: bounded updates + rationale             │    no sibling scores)
   │          via forced tool call                    │
   └──────────────────────────────────────────────────┘
         │  child program (validated: unique, memory-safe)
         ▼
   ┌──────────────────────────────────────────────────┐
   │  executor.run()   [Complexa — v2/executor.py]    │   the expensive step
   │  InferenceProgram → Hydra overrides → subprocess │   (22–153 min / eval)
   │  → sequential stage execution → rewards CSV      │
   │  → ProgramScore                                  │
   └──────────────────────────────────────────────────┘
         │  ProgramScore (n_successful, n_finals, …)
         ▼
   ┌──────────────────────────────────────────────────┐
   │  tree.backup()   [PUCT — v2/tree.py]             │   pure tree search
   │  propagate score up to root; child's rank_key    │
   │  is primary_successes (count, not pass rate)     │
   └──────────────────────────────────────────────────┘
         │
         └─── loop back to tree.select() until
              budget_evals or wall-clock exhausted

Where human design actually enters

All of the following are hand-picked by the researcher and are fixed for all four runs on this page:

  1. InferenceProgram schema — the data classes in autoresearch_v2/inference_program.py define every knob that can exist.
  2. Tier-A mutation surface — the subset the LLM is allowed to change. The exact allowed paths depend on mutator scope and campaign settings.
  3. Seed programs — balanced, deep, wide, notto, noise020, halluc_greedy5, mixed_noise_halluc, mcts_msg5, mcts_rws5, and ∼20 more. Each is a concrete hypothesis about how to attack a target. This page's campaign used one fixed seed profile per run.
  4. Routing map — which seed is picked for which target. 05_CD45 → noise020, 27_HER2_AAV → halluc_greedy5, etc. This map is the audited page's Figure H.
  5. Memory cap — the formula beam_width × nsamples × batch_size ≤ 512 and its threshold of 512.
  6. Success threshold — paper-aligned gates af2folding_i_pae < 7/31 AND af2folding_plddt_log > 0.9.
  7. Idea bank — hand-written one-sentence hypothesis hints (seed_diversify_early, etc.) supplied with every LLM call.
  8. PUCT constants — c_puct = 1.4, children_per_expansion = 2, virtual-loss weight.
  9. Evaluation metrics — the definitions of pass_rate = n_successful / n_finals (online) and the post-hoc unique_successes via ProteinMPNN redesign + ESMFold + FoldSeek.

What is chosen during a run

  1. The next child mutation → LLM mutator
  2. The next parent to expand → PUCT tree search
  3. Visit allocation across the tree → PUCT tree search
  4. The executed child program sequence → determined by the stored select → mutate → execute → backup loop

Role split. In these runs, seed choice, mutation scope, idea bank, thresholds, and memory cap are fixed before execution. Parent selection is performed by PUCT, and field updates are proposed by the LLM mutator.

5. Methodology and Budget

Job layout

Four SLURM jobs, each self-hosted in a single container:

  • GPU 0 hosts a local vLLM server (Qwen3.6-35B-A3B-FP8 or Gemma-4-E4B)
  • GPU 1 runs the autoresearch_v2 PUCT driver which calls the server over 127.0.0.1
  • BUDGET_EVALS = 20, BUDGET_HOURS = 10, seed profile = balanced
  • Qwen3.6 runs with enable_thinking=true per request; reasoning is captured to llm_interactions/*.jsonl alongside every proposed mutation

Mutation surface

For the campaign on this page, the LLM receives a parent InferenceProgram and the Tier-A mutation surface: search algorithm + its per-algorithm params, sampling knobs (guidance_w, ag_ratio, n_recycle, self_cond), refinement settings, filter, and dataloader. It emits bounded updates via a forced tool call, staying under the memory cap beam_search.beam_width × stage.nsamples × stage.batch_size ≤ 512.

How the chains below were produced

For each run, the best node is picked by primary_successes (count). Parents are then walked up to the root to give the ordered chain of programs shown. The mutation field for each row is the diff between parent and child programs read directly from the stored programs/*.json—not the LLM's prose “description,” which can overstate or understate what actually changed.

What the LLM's mutation_description field says (verbatim)

  • gen 1 (bw 8→10): "Increasing beam_width from 8 to 10 expands the early exploration frontier to test the diversity hypothesis more conservatively, avoiding the failures observed at width=16."
  • gen 2 (soft_opt + n_temp_iters): "Enabling soft optimization and increasing temperature annealing steps from 45 to 75 aims to improve local structural relaxation of top candidates without expanding the search budget."
  • gen 3 (bw 10→12): "Increasing beam_width to 12 expands the early exploration frontier to capture more diverse candidates."

  • gen 5 (ns 8→16, bs 4→2): "Quadrupling the total sampling budget… maintains memory safety at 384 while testing if further increasing early-stage diversity pushes the near-ceiling success rate closer to 100 %."
  • gen 6 (bw 12→16): "Increasing the beam width to 16 expands the early-stage exploration frontier to capture rare high-affinity topologies, keeping the memory product exactly at the 512 GPU limit."

Chain-of-thought trace from the LLM call that produced gen 1 (verbatim, bottom of thinking window)

However, the hint explicitly mentions beam_width/nsamples. I'll stick to beam_width=10.

Actually, looking at the prior run feedback: "Increasing beam_width to 16 expands the early
exploration budget to identify ..." It failed completely. This might be due to OOM or just bad
luck. A step to 10 is logical.

Final check of the proposal:
- updates: [{"path": "search.beam_search.beam_width", "value": 10}]
- mutation_description: "Increasing beam_width from 8 to 10 expands the early exploration
  frontier to test the diversity hypothesis more conservatively, avoiding the failures observed
  at width=16."
- selected_idea_ids: ["seed_diversify_early"]

All constraints met. Proceeds.

Two details visible in this excerpt and in the stored thinking:

  • The accepted gen 1 call references a prior 0-success sibling at beam_width=16 and then proposes beam_width=10.
  • The same excerpt recomputes the memory product before the tool call (10 × 4 × 8 = 320 ≤ 512).

Chain-of-thought trace from the LLM call that produced gen 4 (verbatim)

[...] So I need to reduce either beam_width or batch_size.

If I keep beam_width=12 and reduce batch_size to 4: 12*8*4 = 384, which is fine.

Alternatively, I could keep batch_size=8 and reduce beam_width to 8: 8*8*8 = 512, which is
exactly at the limit.

Let me try increasing nsamples while adjusting batch_size to stay within memory limits. This
tests the hypothesis that more samples at the same beam width yields more successes.

In the stored trace for gen 4, the accepted call keeps beam_width fixed and changes nsamples and batch_size, giving a memory product of 384 while doubling total samples.

Pass-rate trajectory

Rows 1–3 move the config from 87.5 % to 100 % at roughly the same scale (n_finals near 40–48). Rows 4–6 then scale up: n_finals goes 48 → 96 → 192 → 256 while pass rate declines from 100 % to 96.9 %.

Gen 5 at the larger pool size (21.1 % → 12.9 %)

13. Observed CD45 Pass-Rate Drop

Both LLM trees show the same pattern on CD45: a first scale-up from 32 to 128 finals lands at 27 successes (21.1 %), and a second scale-up from 128 to 256 finals adds only 6 more successes (33 total, 12.9 %). The marginal pass rate of the additional 128 finals is 6 / 128 = 4.7 %.

Scope of this page. The current page establishes the observed count and pass-rate change only. It does not identify the cause of that change.

The same pattern appears in the Gemma/CD45 chain (27 / 128 = 21.1 %, then 33 / 256 = 12.9 %).

14. Caveats and What This Is NOT

Online proxy only; not the audited paper metric

Every pass rate and count on this page is from n_successful / n_finals over raw Complexa finals using the threshold af2folding_i_pae < 7/31 AND af2folding_plddt_log > 0.9. The website's audited results page uses ProteinMPNN redesign + ESMFold refolding + scRMSD_ca < 1.5 Å + FoldSeek clustering. That pipeline has not been run on these outputs, so these numbers are not directly comparable to the audited human-baseline columns (e.g. Deep / Self in Section 4 of the audited page).

Count vs pass rate select different nodes

PUCT ranks nodes by primary_successes (count). In these runs, the node with the highest pass rate appears earlier than the node with the highest count. Comparisons should therefore note both n_finals and pass rate.

Single seed per target, single run per cell

Every tree started from balanced. No variance estimate across seeds, restarts, or LLM sampling temperatures. The best chains above are one realization of a stochastic search.

Gemma/CD45 hit SLURM wall-clock

Job 7127157 was terminated at 12 h before its 20-eval driver budget was exhausted (9 nodes scored). The other three runs completed within their driver budget.

15. Recorded Follow-Up Items

  1. Matched-budget routing baseline (SLURM jobs 7148364 / 7148365). See Section 5. Result recorded on this page: human v1 deep hits 100 % on PDL1, and noise020 hits 25 % on CD45. The logged tree runs on this page remain below the human baseline on matched-scale CD45 pass rate.
  2. Run the post-hoc audit pipeline on the top-3 nodes of each tree to populate paper_audited_metric (FoldSeek unique successes). Only after this are head-to-head comparisons against the audited human baselines meaningful.
  3. Fixed-budget v2 rerun. Cap beam_width × nsamples at a constant (e.g. 32) so the LLM can only reshape rather than resize the final pool, then re-score on pass rate.
  4. Seed-profile sweep. Rerun Qwen/PDL1 and Qwen/CD45 from deep, wide, notto roots to measure start-config sensitivity.
  5. Dead-expansion detector. Teach the mutator's feedback blob to flag when a child produced identical n_successful and n_finals to its parent.
